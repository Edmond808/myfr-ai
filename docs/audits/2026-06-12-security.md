## Top findings (P0/P1)

### 1. Critical — `dispatch_job` RPC lacks caller authorization

**Files:** `schema.sql` (L180–204), `supabase/migrations/004_promoted_and_analytics.sql` (L37–61), `supabase/00_RUN_THIS_IN_SUPABASE.sql` (L191–215), `scripts/check-supabase.mjs` (L61–68)

**Issue:** `dispatch_job(p_job_id)` is `SECURITY DEFINER` and does **not** verify `auth.uid()` owns the job (contrast `accept_quote`, which does). Supabase exposes RPCs to the `anon` and `authenticated` roles by default; `check-supabase.mjs` invokes it with the anon key successfully.

**Impact:** Any client with the public anon key can call `supabase.rpc('dispatch_job', { p_job_id })` for arbitrary job UUIDs — re-dispatching jobs, spamming verified merchants with duplicate pending quotes, and flipping job status to `dispatched`.

**Recommended fix:**

```sql
-- Inside dispatch_job, before INSERT:
if not exists (
  select 1 from jobs where id = p_job_id and customer_id = auth.uid()
) then
  raise exception 'unauthorized';
end if;

-- Also add SET search_path = public (see finding #8)
revoke execute on function dispatch_job(uuid) from anon;
-- Keep authenticated + service_role, or restrict to service_role only
-- and call dispatch only from a service-role server path.
```

Prefer calling `dispatch_job` only from a **service-role** server client after job insert, not from the user's session.

### 2. High — `/api/classify` unauthenticated with no rate limiting

**Files:** `app/api/classify/route.ts`, `lib/classify.ts` (L106–172, server-side key usage only)

**Issue:** `POST /api/classify` accepts anonymous requests with no IP/user rate limit, no request-size cap beyond JSON parse, and no CAPTCHA. When `ANTHROPIC_API_KEY` is set, each request triggers a paid Anthropic call.

**Impact:** Cost exhaustion / DoS against the classification API (OWASP API4: Unrestricted Resource Consumption). Anonymous classify is **intentional** per `MIGRATION_PLAN.md` Phase 2 — the gap is missing abuse controls.

**Recommended fix:**

- Add edge rate limiting (e.g. Vercel KV / Upstash: 10–20 req/min per IP).
- Cap `text` length in the route (e.g. 2000 chars, aligned with `/api/jobs`).
- Return `429` with `Retry-After` header.
- Optional: separate stricter limits when `source === "ai"`.

---

### 3. High — Verified merchant contact PII exposed via RLS

**Files:** `schema.sql` (L117–121), `merchants` table (L38–39)

**Issue:** Policy `"verified merchants visible"` grants `SELECT` on **all columns** for `status = 'verified'` to any role (including `anon`). Quote cards only need `business_name`, `rating`, `jobs_completed`, promotion fields — not `email` or `whatsapp_phone`.

**Impact:** Any authenticated or anonymous client can scrape merchant email and WhatsApp numbers via direct Supabase queries.

**Recommended fix:**

- Replace broad table policy with a **public view** exposing only safe columns, or split policies per column using restricted views.
- Example: `create view merchants_public as select id, business_name, rating, jobs_completed, is_promoted, promotion_rank, promotion_expires_at from merchants where status = 'verified'`.

---

### 4. High — `POST /api/jobs` trusts client-supplied classification

**Files:** `app/api/jobs/route.ts` (L31–59, L72–96), `lib/api-client.ts` (L22–37)

**Issue:** Dispatch validates enum shapes and string lengths but does **not** re-run classification server-side. A logged-in user can POST arbitrary `classification` (wrong category, location, budget) and trigger `dispatch_job` to unrelated merchants.

**Impact:** Merchant spam, category mis-routing, integrity bypass of core IP pipeline.

**Recommended fix:**

- Re-classify `rawRequest` server-side in `POST /api/jobs` (call `classifyRequest(rawRequest, classification.location)` internally) and compare or replace client payload.
- Alternatively require a signed classification token from `/api/classify` (HMAC + expiry) presented at dispatch.

---

### 5. High — Profile loyalty tier self-escalation via API

**Files:** `app/api/profile/route.ts` (L33–44), `schema.sql` (L114–115, L25)

**Issue:** `PATCH /api/profile` passes the full JSON body to `.update(body)`. TypeScript types only document three fields, but extra keys (e.g. `loyalty_tier: 3`) are not stripped. RLS policy `"own profile"` allows `FOR ALL` on own row.

**Impact:** Customers can grant themselves top loyalty tier and associated UI benefits.

**Recommended fix:**

```typescript
const allowed = pick(body, ["full_name", "preferred_language", "default_location"]);
await supabase.from("profiles").update(allowed).eq("id", user.id);
```

Add a DB trigger or restrict `loyalty_tier` updates to `service_role` only.

## Medium findings

### 6. `merchant_job_feed` view may bypass RLS

**Files:** `schema.sql` (L167–175), `app/pro/dashboard/page.tsx` (L32–36), `app/api/merchants/feed/route.ts`

**Issue:** Postgres views owned by `postgres` run with owner privileges and **bypass RLS** on underlying `jobs`/`quotes` unless created with `security_invoker = true` (PG15+). A browser client could query `merchant_job_feed` without the API route's merchant filter and read all dispatched jobs.

**Recommended fix:** Recreate view with `security_invoker = true`, or replace with an RPC that enforces `merchant_id = auth.uid()` merchant lookup.

---

### 7. Analytics endpoint allows unauthenticated write spam

**Files:** `app/api/analytics/quote-filters/route.ts`, `supabase/migrations/004_promoted_and_analytics.sql` (L30–31)

**Issue:** `POST /api/analytics/quote-filters` does not require auth. RLS policy `"log quote filter preferences"` uses `with check (true)` — anyone can flood `user_quote_preferences`.

**Recommended fix:** Require auth for logged-in flows; rate-limit anonymous inserts by IP; add RLS `with check (auth.uid() = user_id OR user_id IS NULL)` plus application-level throttling.

---

### 8. `SECURITY DEFINER` functions missing `SET search_path`

**Files:** `dispatch_job`, `accept_quote` in `schema.sql` / `002_accept_quote.sql` (compare `handle_new_user` in `005_fix_handle_new_user.sql` which sets `search_path = public`)

**Issue:** Without pinned `search_path`, search-path hijacking can affect definer functions.

**Recommended fix:** Add `set search_path = public` to all `SECURITY DEFINER` functions.

---

### 9. Password login open redirect (`next` param)

**Files:** `components/auth/LoginForm.tsx` (L18, L53), `components/auth/RegisterForm.tsx` (L18, L71)

**Issue:** OAuth callback hardens `next` (`app/auth/callback/route.ts` L8–9: blocks `//`), but password login uses `router.push(next)` without validation — e.g. `?next=https://evil.com`.

**Recommended fix:** Share the same `sanitizeNext()` helper used in the callback; reject external URLs.

---

### 10. No security response headers

**Files:** `next.config.ts`

**Issue:** No `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, or `Strict-Transport-Security`.

**Recommended fix:** Add `headers()` in `next.config.ts` or middleware.

---

### 11. Realtime quote channel scoped only by `job_id`

**Files:** `components/RivlyApp.tsx` (L315–338)

**Issue:** Subscription filter is `job_id=eq.${id}` without server-side channel authorization. Supabase Realtime should respect RLS on `quotes`; verify Realtime RLS is enabled in project settings. If misconfigured, quote updates leak across customers.

**Recommended fix:** Confirm Realtime RLS enabled; consider private channels or polling-only until auth is verified.

## RLS policy summary

| Table | Policies | Concern |
|-------|----------|---------|
| `profiles` | Own row ALL | Over-permissive UPDATE (loyalty tier) |
| `merchants` | Owner ALL + verified SELECT all columns | PII over-exposure |
| `jobs` | Customer ALL + merchant SELECT/UPDATE via quotes | OK |
| `quotes` | Customer read/update + merchant ALL | OK; `accept_quote` RPC adds atomicity |
| `transactions` | Customer/merchant SELECT | OK |
| `user_quote_preferences` | INSERT always true | Spam vector |

**`accept_quote` RPC:** Correctly checks `customer_id = auth.uid()` and quote status before atomic accept/reject — **good pattern** to replicate for `dispatch_job`.
